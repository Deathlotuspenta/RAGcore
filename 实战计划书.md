# 实战计划书：个人笔记知识库系统

> **跟学完的区别**：这份计划不让你做孤立练习。每个 Phase 你都在 **为同一个项目添砖加瓦**，最后拿到的是一个完整可用的产品，不是一堆扔掉的实验脚本。

---

## 项目定位

上传你的笔记（Markdown / PDF / TXT），AI 自动理解内容，你可以用自然语言提问，得到带来源引用的回答。知识库是**动态的**——随时加新笔记、删旧笔记、重新索引。

## 技术栈一览

| 组件 | 用什么 | 为什么 |
|------|--------|--------|
| LLM | DeepSeek（主）/ OpenAI（备） | 配置化切换，你已经会用 |
| Embedding | BGE-small-zh-v1.5（本地免费） | 中文友好，不花钱 |
| 向量库 | ChromaDB（嵌入式） | 零配置，像 SQLite 一样 |
| 后端 | FastAPI + SSE | 异步 + 流式 |
| 前端 | 单页 HTML + 原生 JS | 不需要学前端框架 |
| 元数据 | SQLite | 笔记信息存在这里 |
| BM25 | `rank_bm25` | 关键词检索，补向量的短板 |
| 文件解析 | PyMuPDF / markdown-it-py | 按格式智能拆 |

### 1.2 `models.py` — 数据模型

```
Note          — id, title, file_type, created_at, chunk_count
```

### 1.3 `embedder.py` — Embedding 抽象层

这是你第一个「抽象层设计」练习：

```python
from abc import ABC, abstractmethod

class BaseEmbedder(ABC):
    @abstractmethod
    def embed(self, texts: list[str]) -> list[list[float]]:
        """返回 List[List[float]]，每个 float list 是一个向量"""
        ...

    @abstractmethod
    def embed_query(self, query: str) -> list[float]:
        """查询向量，单条"""
        ...
```

**你需要实现两个子类**：

`BGEEmbedder`：
- 用 `sentence_transformers` 加载 `BAAI/bge-small-zh-v1.5`
- BGE 模型对 queries 需要加 `"Represent this sentence for searching relevant passages: "` 前缀（这是 BGE 的特殊要求）
- 文档不需要前缀

`OpenAIEmbedder`：
- 用 `openai.OpenAI().embeddings.create()` 调 `text-embedding-3-small`
- 注意 OpenAI embedding API 的输入格式差异

### 1.4 `parser.py` — 文件解析

解析不同格式，提取纯文本内容：

**Markdown**（用 `markdown-it-py` 或正则）：
- 提取纯文本（去掉 `#` `**` `[]()` 等标记符号）
- 但保留标题信息——标题可以作为 chunk 的元数据

**PDF**（用 `pymupdf`，即 `fitz`）：
- 逐页提取文本
- 去掉页眉页脚噪音

**TXT**：
- 直接读

**函数签名**：`def parse(file_bytes: bytes, file_type: str) -> str` 返回纯文本

### 1.5 `chunker.py` — 分块

**核心函数**：`def chunk(text: str, chunk_size: int, overlap: int) -> list[dict]`

每个 chunk 返回 `{"content": "...", "metadata": {...}}`

**实现三种策略**，按格式自动选：

| 策略 | 怎么拆 | 适用格式 |
|------|--------|----------|
| 段落边界 | 按 `\n\n` 分割，合并短段落直到接近 chunk_size | Markdown, TXT |
| 固定大小 | 字符数达到 chunk_size 就切，保留 overlap | 所有（fallback） |
| 标题层级 | 按 `#` `##` `###` 分割，每个标题段独立成 chunk | Markdown（优先） |

**你要处理的情况**：
- 单段超过 chunk_size → 切到句子边界（`。！？\n`）
- 空段落跳过
- overlap 指和前一个 chunk 重叠的字符数

### 1.6 `kb_manager.py` — 知识库管理

这是最核心的类。它管两样东西：
- **SQLite**：笔记元信息（id, title, file_type, created_at）
- **ChromaDB**：每个 chunk 的向量 + 文本

```python

        ...

    def add_note(self, title: str, content: str, file_type: str) -> str:

        1. 分块
        2. 嵌入所有 chunks

        1. 从 SQLite 删除

        ChromaDB 的删除方法是 collection.delete(where={"note_id": note_id})

    def list_notes(self, page: int, size: int) -> dict:
        """分页列出笔记。返回 {items: [...], total: N, page: 1, size: 20}"""
        ...

        """获取单个笔记详情，包含所有 chunks"""
        ...
```

# Phase 2：检索升级 — 混合检索 + Rerank（目标 ~3h）

> **这个 Phase 做完**：搜索质量质变。关键词查询不会再漏掉。

## 你要实现的东西

### 2.1 `retriever.py` 升级 — BM25 + RRF

**BM25 是什么**：传统的「关键词匹配」算法。向量检索找的是「语义相似」，BM25 找的是「词匹配」。两者互补：

- 搜「Docker 部署」→ 向量能找到「容器化上线流程」（语义相似）
- 搜「Docker 部署」→ BM25 精确命中「Docker 部署步骤」那段（词匹配）

**RRF（Reciprocal Rank Fusion）**：融合两路结果的标准方法。

```
RRF_score(d) = Σ  1 / (k + rank_i(d))
```

- `k = 60`（经典默认值）
- 意思是「排在前面比排在后面重要得多」
- 对每个文档：它在向量结果里排第几？在 BM25 结果里排第几？两个分数加起来就是最终排名

**你要做的**：
1. 把所有 chunks 喂给 `BM25Okapi`（from `rank_bm25`），建一个关键词索引
2. 每次查询同时跑向量检索和 BM25
3. 用 RRF 融合两路结果，取融合后的 top_k

### 2.2 `reranker.py` — 重排序

检索回来的 10 条只是「可能相关」。Reranker 对每条精排，留下最相关的 5 条。

**本地方案**（推荐先做）：用 `FlagEmbedding` 的 `bge-reranker-v2-m3`：
```python
from FlagEmbedding import FlagReranker
reranker = FlagReranker('BAAI/bge-reranker-v2-m3')
scores = reranker.compute_score([[query, chunk] for chunk in chunks])
```

**API 方案**（备选）：Cohere Rerank API

### 2.3 `/search` 端点升级

加一个 `retrieval_mode` 参数：
- `"vector"` — 纯向量
- `"hybrid"` — 向量 + BM25
- `"hybrid+rerank"` — 向量 + BM25 + Reranker（默认）

返回结果里每条标注 `retrieval_method`。

## Phase 2 检查：同一个搜索词，三种模式对比，hybrid+rerank 应该最好

# Phase 3：RAG 生成 + Web UI（目标 ~4h）

> **这个 Phase 做完**：浏览器打开就能用——上传笔记、提问、看流式回答、点来源跳转。

## 你要实现的东西

### 3.1 `generator.py` — LLM 生成层

这是你第二个「抽象层设计」练习，和 `embedder.py` 模式一样：

```python
class BaseGenerator(ABC):
    @abstractmethod
    async def generate(self, messages: list[dict]) -> str:
        """非流式生成"""
        ...

    @abstractmethod
    async def generate_stream(self, messages: list[dict]) -> AsyncIterator[str]:
        """流式生成，每次 yield 一个 token"""
        ...
```

**DeepSeekGenerator**：调 DeepSeek chat API，`stream=True` 时用 `httpx` 的 iter_lines 逐行解析 SSE。

**OpenAIGenerator**：用 `openai` SDK 的 `stream=True`。

### 3.2 RAG Prompt 模板

```
System: 你是一个个人知识库助手。根据以下检索到的笔记内容回答用户问题。
如果笔记中没有相关信息，就说"我的笔记中没有相关记录"，不要编造。
每条回答必须标注信息来源。

检索到的笔记内容：
{context}

用户问题：{query}
```

**你要处理**：
- 检索结果太长 → 只取 top 5 的 content 拼接
- 拼接后超出 context window → 截断最早的 chunk

### 3.3 API 端点

**`POST /ask`** — 非流式问答：
```json
// Request
{"query": "什么是 RAG？"}
// Response
{
  "answer": "RAG（Retrieval-Augmented Generation）是...",
  "sources": [
    {"note_title": "RAG 入门.md", "chunk_id": "xxx", "content": "..."},
    ...
  ]
}
```

**`GET /ask/stream`** — SSE 流式：
```
event: status
data: {"stage": "retrieving", "message": "正在检索相关笔记..."}

event: status
data: {"stage": "generating", "message": "找到 5 个相关片段，正在生成回答..."}

event: token
data: {"token": "RAG"}

event: token
data: {"token": "（"}

...  // 逐 token

event: done
data: {"sources": [...]}
```

### 3.4 Web UI

单页 HTML（`static/index.html`），分左右两栏：

**左栏 — 知识库管理**：
- 拖拽/点击上传区（支持 .md / .pdf / .txt）
- 笔记列表（标题 + 日期 + 删除按钮）
- 上传进度条

**右栏 — 对话**：
- 聊天界面（问题输入 + 答案流式显示）
- 每条回答下面可展开「📎 来源引用」
- 点击来源高亮对应的 chunk 内容

**纯原生 JS 实现**，不引入 React/Vue。用 `EventSource` 接收 SSE 流式回答。
